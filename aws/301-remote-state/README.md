# AWS Remote State Terraform

A minimal **Terraform** configuration that stores state in an S3 bucket with DynamoDB locking. Bootstrap scripts create the backend resources in AWS; Terraform reads the backend settings directly from `backend.tf`.

This example uses Terraform's native remote state backend. It is **not** a Terragrunt setup — there is no `terragrunt.hcl`, no generated backend config, and no `terragrunt` commands.

## Purpose

This example builds on [101-basic](../101-basic/). It shows:

- How to configure the native Terraform `s3` remote state backend in `backend.tf`
- How to bootstrap backend storage with AWS CLI scripts before running Terraform
- How Terraform manages infrastructure when state is stored remotely

The state backend (S3 bucket, DynamoDB lock table) is created outside Terraform by the bootstrap scripts. Terraform itself only manages the sample workload S3 bucket.

## Prerequisites

- [tenv](https://tofuutils.github.io/tenv/) for Terraform version management — see [tenv.md](../../tenv.md)
- Terraform `>= 1.15.0` (pinned to **1.15.7** via `.terraform-version`)
- [AWS provider](https://registry.terraform.io/providers/hashicorp/aws/latest) `~> 5.0`
- [AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html) installed and configured

Install and select the pinned Terraform version:

```bash
tenv tf install    # reads .terraform-version
terraform version  # should report 1.15.7
```

Configure AWS credentials before running the bootstrap scripts or Terraform:

```bash
aws configure
```

## Project structure

```
aws/301-remote-state/
├── backend.tf              # Remote state backend configuration
├── bootstrap-backend.sh    # Create backend storage (Bash)
├── bootstrap-backend.ps1   # Create backend storage (PowerShell)
├── providers.tf            # Terraform version, provider requirements, and AWS provider
├── variables.tf            # Input variables
├── main.tf                 # Sample workload resource
├── outputs.tf              # Output values
└── README.md               # This file
```

| File | Responsibility |
|------|----------------|
| `backend.tf` | Native Terraform `s3` backend settings |
| `bootstrap-backend.sh` | Prompts for profile, generates bucket name, updates `backend.tf` (Bash) |
| `bootstrap-backend.ps1` | Prompts for profile, generates bucket name, updates `backend.tf` (PowerShell) |
| `providers.tf` | Pins Terraform and the `aws` provider |
| `variables.tf` | Defines configurable inputs with defaults |
| `main.tf` | Declares the sample workload S3 bucket |
| `outputs.tf` | Exposes useful values after apply |

## What gets created

### By bootstrap scripts (remote state backend)

| Resource | Name (default) | Purpose |
|----------|----------------|---------|
| S3 bucket | `learn-terraform-state-{account-id}-` + 9 random digits | Stores Terraform state objects |
| DynamoDB table | `learn-terraform-state-lock` | State locking |

The bootstrap scripts ask which AWS profile to use (if multiple exist), generate a globally unique bucket name, update `bucket` in `backend.tf`, and create the matching AWS resources.

### By Terraform (workload)

| Resource | Name (default) | Region (default) |
|----------|----------------|------------------|
| S3 bucket | `{account-id}-learn-terraform-workload` | `eu-north-1` |

## Variables

| Name | Type | Default | Description |
|------|------|---------|-------------|
| `region` | `string` | `eu-north-1` | AWS region for the workload S3 bucket |
| `bucket_name_suffix` | `string` | `learn-terraform-workload` | Suffix appended to the account ID for the workload bucket name |

## Outputs

| Name | Description |
|------|-------------|
| `bucket_name` | Name of the workload S3 bucket |
| `bucket_arn` | ARN of the workload S3 bucket |

## Usage

### Step 1 — Bootstrap the AWS storage backend

Run one of the bootstrap scripts before `terraform init`. The script:

1. Lists your AWS profiles and asks which one to use (if multiple exist)
2. Generates a unique state bucket name like `learn-terraform-state-123456789012-342254543`
3. Updates `bucket` in `backend.tf`
4. Creates the S3 bucket (versioning, encryption, public access block) and DynamoDB lock table

**Bash:**

```bash
cd aws/301-remote-state
./bootstrap-backend.sh
```

**PowerShell:**

```powershell
cd aws/301-remote-state
./bootstrap-backend.ps1
```

Example prompt:

```text
Available AWS profiles:

  1) default
  2) sandbox

Select profile [1-2]: 2

Using AWS profile: sandbox

Generated state bucket name: learn-terraform-state-123456789012-342254543
Updated bucket in backend.tf
```

Other backend settings (`region`, `dynamodb_table`, `key`) come from `backend.tf`. Only `bucket` is generated by the bootstrap scripts.

### Step 2 — Run Terraform

```bash
terraform init
terraform plan
terraform apply
```

Use the `terraform` CLI directly — not `terragrunt`. Terraform stores its state in the S3 bucket configured in `backend.tf`.

### Cleanup

Remove workload resources first:

```bash
terraform destroy
```

Then delete the backend storage manually:

```bash
aws s3 rb s3://YOUR-STATE-BUCKET-NAME --force
aws dynamodb delete-table --table-name learn-terraform-state-lock --region eu-north-1
```

Replace the bucket name with the value from `backend.tf`.

## Authentication

The bootstrap scripts and Terraform use credentials from the AWS CLI or environment variables by default.

For CI/CD, use IAM roles or OIDC with permissions on the state bucket and DynamoDB table. See [S3 backend authentication](https://developer.hashicorp.com/terraform/language/settings/backends/s3).
